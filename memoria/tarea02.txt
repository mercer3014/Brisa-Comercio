CONTEXTO
--------
El sistema Geodata ya esta funcionando completo (portal publico: Inicio, Explorar,
Rankings, Acerca de; y panel de administracion con todos sus modulos). Funciona bien,
pero el DISENIO actual (azul generico, plano) no convence. Esta tanda tiene dos metas:

  PARTE A - Rediseniar la identidad visual (portal publico, panel y login) hacia un
            estilo mas editorial, institucional y moderno.
  PARTE B - Preparar el proyecto para que sea reproducible: pasar la base de datos a
            MIGRATIONS de Laravel y los datos base (permisos, roles, usuario admin) a
            SEEDERS, para poder levantar todo con un comando.

Mantener las reglas permanentes: carpeta memoria/ (leer al iniciar, actualizar al
terminar), codigo y textos en espaniol, no romper funcionalidad existente.

DIRECCION DE DISENIO (importante, leer con atencion)
----------------------------------------------------
El estilo objetivo combina dos referencias:
1. Un estilo EDITORIAL OSCURO para el portal publico: fondo azul institucional muy
   oscuro en las secciones principales (hero), titular grande tipo titular de prensa,
   un panel de "resumen del periodo" flotante con las cifras clave, y tarjetas de
   indicadores limpias debajo. Sensacion de "portal de datos serio y profesional".
2. Un header y una estructura general mas trabajados y modernos (inspiracion: sitios
   corporativos amplios, con menu claro, sensacion de espacio, sin saturar), usando
   imagenes de fondo donde aporten.

PALETA INSTITUCIONAL OBLIGATORIA (escudo FCEE - UAGRM):
- Azul institucional:  #193153  (color dominante, estructura, fondos oscuros del hero)
- Rojo institucional:  #e10f1c  (ACENTO, en dosis pequenias: subrayado del titular,
                                 item de menu activo, boton principal, dato resaltado)
- Blanco:              #ffffff  (contenido y tarjetas claras)
- Derivar: un azul mas oscuro para fondos del hero (ej. casi navy), grises auxiliares
  para textos secundarios y fondos alternos, verde sobrio para variaciones positivas y
  el rojo institucional para variaciones negativas. Buen contraste (accesibilidad AA).

Regla de oro del color: el AZUL manda, el ROJO es acento puntual. Nada de grandes
areas rojas. El portal debe verse institucional y serio, no estridente.

==============================================================================
 PARTE B - PREPARACION DEL PROYECTO (migrations y seeders)
==============================================================================

CONTEXTO: hoy las tablas se crearon ejecutando un script SQL a mano, y los datos base
(roles, permisos, usuario admin) se sembraron sueltos. Para que el proyecto sea
reproducible y profesional, hay que pasar todo a migrations y seeders de Laravel.

IMPORTANTE: hacerlo con cuidado para no perder los datos ya cargados en la base de
desarrollo. Coordinar con el equipo: lo ideal es validar las migraciones contra una base
NUEVA y vacia, y dejar la base actual intacta hasta confirmar que todo reproduce igual.


TAREA B1 - Migraciones de todo el esquema
-----------------------------------------
OBJETIVO: que las 39 tablas del sistema se puedan crear con "php artisan migrate".
ESPECIFICACIONES:
1. Crear una migracion por tabla (o agrupadas por modulo) que reproduzca EXACTAMENTE la
   estructura actual: tipos, llaves primarias (identity), llaves foraneas, restricciones
   UNIQUE y CHECK, indices, la columna JSONB atributos_extra y el indice GIN.
   Referencia: el archivo estructura-bd.sql en la raiz del proyecto. Respetar nombres
   EXACTOS de tablas y columnas.
2. Respetar el orden de dependencias (tablas referenciadas antes que las que las usan; la
   tabla de hechos al final). Para la FK de operacion_comercio_exterior hacia
   carga_archivo, crear esa FK despues si hace falta, igual que en el script.
3. Mantener las tablas internas de Laravel (jobs, cache, sessions) en sus propias
   migraciones.
4. Verificar ejecutando "php artisan migrate:fresh" sobre una base de prueba vacia: debe
   crear las 39 tablas sin error y quedar identica a estructura-bd.sql.
CRITERIOS DE ACEPTACION:
- "php artisan migrate" levanta todo el esquema desde cero en una base vacia.
- La estructura resultante coincide con la actual (mismos nombres, tipos y relaciones).


TAREA B2 - Seeders de datos base
--------------------------------
OBJETIVO: sembrar los datos minimos para que el sistema arranque utilizable.
ESPECIFICACIONES:
1. Seeder de ROLES: administrador, analista, consultor, invitado.
2. Seeder de PERMISOS: todos los codigos de permiso usados por el sistema (los mismos que
   ya existen: bitacora.ver, calidad.tratar, calidad.ver, carga.crear, carga.procesar,
   carga.ver, catalogo.editar, catalogo.ver, configuracion.editar, configuracion.ver, y
   los demas que el panel de Roles ya muestra). Revisar el sistema para listarlos todos.
3. Seeder de ROL-PERMISO: asignar al rol administrador TODOS los permisos; a analista,
   consultor e invitado los que correspondan (analista: cargas, ETL, calidad, catalogos;
   consultor: ver/explorar/dashboards/reportes/exportar; invitado: solo lectura basica).
4. Seeder de USUARIO ADMIN: crear un usuario administrador inicial con su contrasenia
   hasheada (bcrypt) y asignarle el rol administrador. Las credenciales por defecto se
   definen en variables de entorno o se documentan claramente para el equipo (y se obliga
   a cambiarla en el primer ingreso mediante el campo debe_cambiar_pwd = true).
5. Seeder de ORGANIZACION: registrar al INE como organizacion 1 (con su fuente_datos y
   los perfiles de mapeo iniciales INE-EXP e INE-IMP si corresponde).
6. Seeder de CONFIGURACION y REGLAS DE VALIDACION: los parametros base (moneda_base,
   sesion_minutos_expira, max_intentos_login, etc.) y las reglas de calidad iniciales.
7. Registrar todos en DatabaseSeeder en el orden correcto, de modo que "php artisan
   db:seed" (o "migrate:fresh --seed") deje el sistema listo para usar.
CRITERIOS DE ACEPTACION:
- Tras "php artisan migrate:fresh --seed" sobre una base vacia, se puede iniciar sesion
  con el usuario administrador sembrado y el sistema queda operativo (roles, permisos,
  INE registrado, configuracion y reglas presentes).

NOTA: NO incluir en seeders los microdatos de comercio (esos se cargan por la interfaz/
ETL). Los seeders son solo catalogos del sistema y datos base.
